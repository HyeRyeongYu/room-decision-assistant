# 1. Business purpose

- 예전부터 오늘날까지 2030세대는 학업과 취업 등의 이유로 자취를 시작하는 경우가 많다. 이 과정에서 원룸, 오피스텔 등의 다양한 주거 공간 중 하나를 선택해야 하며, 주거 환경을 직접 확인하고, 계약을 진행하는 과정에서 여러 요소에 관한 판단과 의사결정 상황에 놓이게 된다. 그러나, 관련 경험과 정보의 부족 그리고 전문 용어에 대한 이해 부족으로 인해 계약 과정에서 어려움을 겪는 경우가 많다. 그 결과, 계약을 체결한 세입자(임차인)는 이전에 충분히 파악하지 못했던 문제를 뒤늦게 발견하고, 이를 해결해야 하는 상황에 직면하기도 한다.
- 특히, 현행 부동산 거래 과정에서는 공인중개사가 계약 체결 직전에 ‘중개대상물 확인·설명서’를 제공할 의무가 있다. (공인중개사법 제25조) 하지만, 소비자가 여러 매물을 탐색하고, 비교하는 초기 단계에서는 공인중개사가 일부 항목을 구두로 설명하는 경우는 있으나, 주거 환경을 체계적으로 기록하고 비교할 수 있는 서면의 형태는 제공하지 않는다. 따라서, 소비자는 인터넷 검색이나 개인적인 경험에 의존하여 체크리스트를 준비하기에, 확인해야 하는 중요 항목을 놓친 채 의사결정을 내리는 문제가 발생하기도 한다. 
- 또한, 기존에 블로그나 유튜브 등의 매체에서 찾을 수 있는 체크리스트는 대부분 여러 매물을 비교할 수 있는 형태가 아닌, 단일 매물을 기준으로 체크리스트가 구성되어 있어, 종합적으로 매물들의 장단점을 비교하고, 판단하는 과정에 한계가 있다. 그리고 최종적으로 결정하기 전까지 여러 매물의 정보를 소비자가 개별적으로 관리하고 보관해야 하는 불편함이 존재한다.
- 따라서, 본 프로젝트에서는 자취방 선택 과정에서 발생하는 문제들을 해결하기 위해, 사용자가 다양한 매물을 체계적으로 평가하고, 비교할 수 있도록 웹 기반의 의사결정 지원 시스템을 개발하고자 한다. 이 시스템은 중개대상물 확인·설명서의 항목과 내용을 바탕으로 체크리스트 기능을 제공하며, 각 매물에 대한 사용자의 평가 결과를 점수로 정량화하여 의사결정의 정확도를 높이고, 중요한 항목의 확인 누락을 방지함으로써, 중개 계약 이후의 리스크를 줄이는 것을 목표로 한다.

# 2. System context diagram

![System Context Diagram](./system-context-diagram.png)

Figure 1. System Context Diagram

### (1) User → System
- Input checklist items : 방(매물) 평가를 위한 체크리스트 항목 입력
- Manage and Select rooms : 여러 방 후보 선택 및 관리
- Record observations : 각 방에 대한 관찰 사항 및 메모 기록
- Assign scores : 체크리스트 기준에 따른 평가 점수 부여
- Request comparison : 서로 다른 방 후보 간의 비교 요청
- Save checklist file : 체크리스트 파일 저장 요청
- Load checklist file : 체크리스트 파일 불러오기 요청

### (2) System → User
- Provide checklist : 법적 가이드라인(중개대상물 확인·설명서)에 기반한 구조화된 체크리스트 제공
- Calculate evaluation scores : 각 방에 대한 항목 평가 점수 계산 제공
- Compare room candidates : 다수의 방 후보 간 비교 결과 제공
- Suggest choice : 사용자 입력 및 평가를 기반으로 최적의 방 추천
- Provide customizable checklist : 사용자 맞춤 체크리스트 항목 제공
- Export checklist data : 체크리스트를 외부 파일로 저장할 수 있도록 제공
- Import checklist data : 외부 체크리스트 파일을 불러올 수 있도록 제공

# 4. Concept of operation
### (1) Add room candidate
| Category | Description |
|:----------:|-------------|
| Purpose | 사용자가 비교를 위해 새로운 방의 체크리스트를 시스템에 등록하기 위함 |
| Approach | 프로그램의 홈 화면에서 체크리스트 추가 기능을 선택하면, 추가하고자 하는 매물의 유형을 선택할 수 있다. 이후 선택된 유형에 맞는 체크리스트 형식이 제공되며, 해당 매물에 대한 기본 정보를 입력할 수 있다. 생성 기능을 선택하면 입력된 정보가 저장되고 새로운 방 후보로 등록된다. |
| Dynamics | 사용자가 방의 체크리스트를 추가하고 싶은 경우 |
| Goals | 사용자가 새로운 방 후보에 대한 체크리스트를 생성할 수 있도록 한다. |

### (2) Evaluate room with checklist  
| Category | Description |
|----------|-------------|
| Purpose | 사용자가 체크리스트 항목을 기준으로 방을 체계적으로 평가하기 위함 |
| Approach | 체크리스트가 제공되면 각 항목에 대해 상/중/하와 같은 평가 기준 중 하나를 선택하여 평가를 수행할 수 있다. 이후, 총점 계산 기능을 선택하면, 평가 결과가 반영되어 총점수가 계산된다. 만약 필수 항목의 평가가 누락된 경우, 누락된 항목의 그룹 번호와 항목 번호가 안내된다. |
| Dynamics | 사용자가 방의 체크리스트를 평가하고자 하는 경우 |
| Goals | 사용자가 매물 계약 체결 전 필수 항목을 누락 없이 확인하고 평가할 수 있도록 한다. |

### (3) Customize checklist

|----------|-------------|

| Dynamics | 사용자가 체크리스트에 항목을 추가 및 평가하고자 하는 경우 |

# 5. Problem statement
## 5.1 Overview
- 본 프로젝트의 목적은 대학생과 사회 초년생이 주거 계약 과정에서 겪을 수 있는 정보 부족 문제와 의사결정의 어려움을 해결하기 위해, 체크리스트 기반 평가 및 비교 기능을 제공하는 웹 기반 의사결정 지원 시스템을 개발하는 것을 목표로 한다.
이를 위해, 다양한 매물 정보를 체계적으로 평가하고 비교할 수 있는 기능을 구현해야 하며, 아래와 같은 문제들을 고려해야 한다.

## 5.2 Problems to be Considered
(1) 체크리스트의 항목 점수 기준 설정 문제
- 체크리스트 기반 평가 시스템을 통해 실제 매물 계약 체결에 도움을 주기 위해서는 각 항목에 대한 정량적인 점수화가 필요하다. 이를 위해 단계별 차등 점수화 방식으로 사용자의 평가를 반영한다. 그러나 항목별 평가 단계를 지나치게 단순화하면 매물의 상태를 충분히 반영하기 어렵고, 반대로 지나치게 세분화하면 사용자의 평가 부담이 증가할 수 있다. 또한 평가 단계가 많아질수록 총점 계산과 매물 비교 과정에서 시스템 구조가 복잡해질 수 있으므로, 적절한 평가 단계 수를 설정하는 것이 중요하다.

(2) 체크리스트 확장 시 구조 문제 (사용자 맞춤 항목 추가)
- 체크리스트의 필수 항목 외에 사용자 맞춤 항목을 추가하기 위해서는 최초 생성 시 만들어지는 기본 체크리스트 데이터와 충돌 없이 사용자가 필요한 시점에 항목 추가할 수 있어야 한다. 그러나 사용자 추가 항목이 과도하게 증가할 경우, 체크리스트의 구조가 복잡해지고, 화면 구성 유지 측면에서 문제가 발생할 수 있다. 또한 기본 체크리스트와 사용자 추가 항목이 충돌 없이 하나의 통합된 체크리스트로 유지될 수 있도록 구조 설계에 대한 고려가 필요하다.

(3) 로컬 스토리지 사용에 따른 데이터 유지 문제
- 본 시스템은 사용자가 생성하고 평가한 체크리스트 데이터를 로컬 스토리지 기반으로 관리한다. 그러나 로컬 스토리지는 동일한 기기, 브라우저 환경에서만 접근이 가능하며, 브라우저 저장 데이터 또는 사이트 데이터가 삭제되는 경우 저장된 정보가 유실될 수 있다. 또한 사용자가 여러 기기에서 동일한 데이터를 활용하거나, 장기간 데이터를 유지하는 데 제한이 발생할 수 있으므로, 데이터 유지 및 관리 측면에서 저장 방식에 대한 추가적인 고려가 필요하다.

(4) 매물 비교 및 추천 기준 설계 문제
- 본 시스템에서는 사용자가 여러 매물의 체크리스트를 평가하고 매물 추천 기능을 요청하면, 시스템은 추천 기준을 통해 매물을 비교하여 사용자에게 의사결정의 정보를 제공한다. 이때 총점수뿐만 아니라, 주거 비용, 사용자 추가 항목 점수 등 다양한 요소가 함께 고려될 수 있어, 비교 기준이 복잡해질 수 있다. 또한 매물 간 동일한 점수가 발생하는 경우 추가적인 비교 기준이 필요할 수 있으며, 사용자가 납득할 수 있으면서 시스템 구조의 안정성을 유지할 수 있는 기준 설계에 대한 고려가 필요하다.

## 5.3 Non-Functional Requirements (NFRs)
(1) 사용성 고려
- 본 시스템은 사용자가 실제 매물을 확인하는 상황에서도 직관적으로 체크리스트의 항목을 평가할 수 있도록 설계되어야 한다. 이를 위해 평가 단계는 이해하기 쉽고 선택하기 편리해야 한다. 또한 복잡한 입력 과정 없이도 빠르게 평가할 수 있도록 화면의 직관성을 고려해야 한다.

(2) 신뢰성 고려
- 사용자가 평가한 체크리스트 데이터가 유실되지 않도록 저장 안정성에 대한 방식을 고려해야 한다. 또한 사용자의 항목 평가 결과를 기반으로 정량적인 점수 계산을 통해 일관된 추천 결과를 제공할 수 있도록 시스템의 신뢰성을 확보해야 한다.

(3) 성능 고려
- 체크리스트의 총점수 계산과 매물 비교 과정에서 지연이 발생하지 않도록 시스템 성능을 고려해야 한다. 또한 정확한 계산을 유지하면서도 빠른 응답을 제공하여, 시스템 사용 과정에서 불편함이 발생하지 않도록 해야 한다.

# 6. Glossary
| Term | Description |
|----------|-------------|
| 매물 | 임대(전세·월세) 등을 목적으로 거래 대상이 되는 주거용 부동산 |
| 사용자 추가 항목 | 기본 체크리스트 외에 사용자가 추가로 입력하는 맞춤 평가 항목|
| 임차 | 일정 비용을 지불하고 일정 기간 동안 타인의 부동산을 사용하는 행위 (↔ 임대) |
| 임차인(세입자) | 법률적 용어로써, 임대차 계약에 따라 보증금이나 월세를 내고 다른 사람의 부동산을 빌려 사용하는 사람 |
| 주거 계약 과정 | 전세와 월세 모두 동일하게 적용되는 표준 절차 : 매물 찾기, 매물 실제 확인 및 권리 분석, 계약 체결, 계약 후 조치 |
| 체크리스트(Checklist) | 확인·설명서 내용을 기반으로 구현된 매물 평가를 위해 제공되는 그룹화된 항목의 목록 |
| 확인·설명서 | 개업 공인중개사가 중개대상물(매물)의 상태, 권리관계, 법적 제한사항 등을 중개 완성 전 확인하여, 거래 당사자(임차인)에게 서면(종이)으로 교부하는 (법적) 문서 |
| Recommendation | 체크리스트 평가 결과를 기반으로 시스템이 사용자에게 제시하는 추천 매물 |
| Room Candidate | 사용자가 평가 및 비교하기 위한 체크리스트에 등록한 매물 |
| Total Score (총점수) | 체크리스트 항목 평가 결과를 기반으로 계산된 전체 점수 |
